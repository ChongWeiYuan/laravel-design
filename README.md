# Laravel-Design

## このリポジトリについて

本リポジトリは、架空の飲食店「ドルドルバーガー(仮名)」の在庫・売上管理システムの構築を
**設計書・要件定義書**から設計～管理～実装するためのものです。

- アプリケーションURL: https://nagao.xyz

### 目的
- **設計プロセスの可視化**: 要件定義→画面設計→DB設計→インフラ設計の各ドキュメントを
  GitHub上で履歴管理し、ポートフォリオとして提示する。
- **AIとの協調開発の記録**: コード生成AI（GitHub Copilot / Codex）を活用した実装の
  前段階として、人間が行った設計判断を記録する。
- **共通フレーム2013の意識**: 実務で用いられるフレームワークを参考にしつつ、
  個人開発であるため法務的な厳密さは省略し、設計の根拠と意思決定に焦点を当てる。

### 設計書類
- 要件定義書
- 画面設計書
- データベース設計書
- インフラ構築手順書
- テスト開発環境手順書

## 開発プロセス

本プロジェクトでは、本番環境とテスト環境をブランチで分離し、以下のフローを採用している。

### ブランチ戦略

| ブランチ | 環境 | URL |
|----------|------|-----|
| `main` | 本番環境 | https://nagao.xyz |
| `staging` | 検証環境 | https://staging.nagao.xyz |
| `develop` | テスト環境 | https://dev.nagao.xyz |
| `feature/*` | ローカル開発 | - |

### ワークフロー

featureブランチで開発 → developにプルリク → テスト環境で動作確認 → mainにマージして本番反映

#### Note:なぜこの開発手法を選択したか

- GitHub Flowをベースに、個人開発でも本番・テストを分離するため
- コード生成AIとの協調開発において、人間のレビューを経てからマージする工程を確保するため
- 実務で標準的なブランチ運用を擬似体験するため
- フォーク開発は個人開発では過剰なため

## 本ドキュメントの位置づけ

本プロジェクトは「共通フレーム2013」に基づいた法務的・契約的なエビデンスとしての厳密な文書化は、下記の理由から省略している。

- ブランク解消のための再学習が主目的である
- AIとの協調開発を前提とした新規学習の場である
- 顧客・発注者が存在しない個人開発である
- 納品後のトラブルや瑕疵担保責任を想定しない

ただし、要件定義・設計・実装の各プロセスは可能な範囲で文書化し、
「設計の根拠」と「実装に至る意思決定の履歴」をGitHub上に残すことを重視する。
